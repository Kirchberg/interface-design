# Interface Design Critique

Critique a built interface like a design lead reviewing for craft, not just correctness.

## Steps

1. Read the relevant UI files or inspect the running UI when available.
2. Read `.interface-design/system.md` if it exists.
3. Read `../interface-design/references/critique.md`.
4. Identify defaulted decisions in:
   - composition and focal point;
   - surface hierarchy;
   - spacing rhythm;
   - typography;
   - content coherence;
   - interaction states;
   - layout structure.
5. Improve the UI directly when the user asked for changes. If the user asked only for critique, report findings without editing.

## Output

For review-only requests, lead with findings and file references.

For implementation requests, make the improvements, verify the result when possible, and summarize what changed.

Do not narrate internal critique steps unless the user asks.
